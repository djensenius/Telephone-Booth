# ADR 0004 — Tailscale serve for debug TLS

**Status:** accepted.

## Context

The booth runs in physical art installations: warehouses, gallery floors,
back rooms with no reverse proxy. Operators want a way to reach the live
debug panel without exposing the Pi to the public internet, without
manually managing certs, and without running a CA. They also want a
working fallback when the Pi is offline from the public internet but
reachable on the LAN.

## Decision

Two cooperating debug listeners, both bearer-token gated:

1. **Tailscale (primary).** The Rust binary listens on `127.0.0.1:8080`
   only. A systemd unit runs `tailscale serve --bg --https=443 / http:
   //127.0.0.1:8080`, publishing the panel at
   `https://booth-N.<tailnet>.ts.net` with a real Let's Encrypt cert
   issued and rotated by Tailscale.
2. **LAN fallback.** The binary also opens `0.0.0.0:8443` with a
   self-signed cert generated by `rcgen` on first boot. The cert's
   fingerprint is written to disk; the operator UI pins it and refuses
   to connect to a different cert without an explicit re-pin.

Either listener can be disabled in config.

## Consequences

**Good:**

- Real LE certs with no DNS or 80/443 exposure.
- A working fallback for LAN debugging when Tailscale is down or
  unavailable to the operator's browser.
- Zero CA infrastructure to run.

**Trade-offs:**

- Tailscale is a dependency — but it's optional; disabling it falls back
  cleanly to the LAN listener.
- Two listeners means two attack surfaces; both gate on the same Bearer
  token and `debug.allow_controls` is the only way to enable mutation
  endpoints.
